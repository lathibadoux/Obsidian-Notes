## 001.3.2.1 Creating Tables

**Bad Practice:
```sql
CREATE TABLE CustomersInfo (
	CustomerID INT, --add primary key
	FirstName, VARCHAR(MAX), --change length to 50, add NOT NULL
	LastName, TEXT, --TEXT type is the worst (use VARCHAR(50), add NOT NULL
	Country VARCHAR(255), --change length to 50, add NOT NULL
	TotalPurchase FLOAT,
	Score VARCHAR(255), --use INT
	BirthDate VARCHAR(255), --use DATE
	EmployeeID INT,
	CONSTRAINT FK_CustomersInfo_EmployeeID FOREIGN KEY (EmployeeID)
		REFERENCES Sales.Employees(EmployeeID)
)

--add non-clustered index on foreign key
```

1. Avoid data types VARCHAR & TEXT.
2. Avoid (MAX) or unnecessarily large lengths in data types.  Takes additional storage and causes SQL to create large indexes.
3. Use the NOT NULL constraint where applicable.  Prevents NULLS from being inserted, so protects the integrity of your data.  However, you also get better index performance.
4. Ensure all your tables have a Clustered Primary Key.
5. Create a Non-Clustered index for foreign keys that are frequently used.

**Good Practice:
```sql
CREATE TABLE CustomersInfo (
	CustomerID INT PRIMARY KEY CLUSTERED, --will default to CLUSTERED 
	FirstName, VARCHAR(50) NOT NULL,
	LastName, VARCHAR(50) NOT NULL,
	Country VARCHAR(50) NOT NULL,
	TotalPurchase FLOAT,
	Score INT,
	BirthDate DATE,
	EmployeeID INT,
	CONSTRAINT FK_CustomersInfo_EmployeeID FOREIGN KEY (EmployeeID)
		REFERENCES Sales.Employees(EmployeeID)
)

CREATE NONCLUSTERED INDEX IX_CustomersInfo_EmployeeID
ON CustomersInfo(EmployeeID)
```
